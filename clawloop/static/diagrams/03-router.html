Router Layer — Model Selection

Routes queries to the cheapest model that passes quality threshold · Learns from both successes and failures

1Query

Incoming Request → Feature Extraction

Extract query features: estimated complexity, token count, domain tags, required capabilities (tool use, code, reasoning). These features determine which tier of models is appropriate.

↓
2Route

Tier Selection → Model Assignment

Models organized in tiers by cost/capability. Router starts from cheapest tier that historically handles similar queries well. If a tier's quality score for this query type is below threshold → escalate to next tier. Fallback chains ensure reliability.

↓
3Score

Track Outcome → Update Per-Model Stats

After episode completes, reward signal updates the model's quality score for this query type. Both successes and failures feed the Router (unlike Harness which only sees failures). Over time, the Router learns which model handles which query type best.

↓
4Optimize

optim_step → Adjust Routing Policy

Periodically recalculate routing thresholds based on accumulated scores. Lower the tier for query types where cheaper models perform well. Raise the tier for query types that need stronger models. Net effect: cost goes down without quality regression.

Example: 3-tier model routing
Haiku

Tier 1 — Fast & Cheap

$0.25/M tokens · 80ms p50
Simple queries, FAQ, classification
Sonnet

Tier 2 — Balanced

$3/M tokens · 400ms p50
Reasoning, multi-step, tool use
Opus

Tier 3 — Maximum

$15/M tokens · 1.2s p50
Complex analysis, code gen, edge cases
